## 1. 方案设计核心原则与应对策略

针对官方给出的约束条件与考查重点，本方案确立了三大核心支柱：

- **完全去 Embedding 化的纯文本结构化索引：** 官方明确禁止使用任何 Embedding 模型进行检索。因此，我们将建立基于 **“目录树/条款编号 + 领域专有名词精确匹配 + 逻辑块（Chunk）”** 的多级倒排索引和结构化标签体系。
- **极致的 Token 经济学（Cost-Aware Routing）：** 盲目将全文喂给模型会导致 Token 爆炸。我们必须采用“漏斗型”过滤机制，先用低成本的规则/轻量模型过滤，再由 Qwen 主力模型进行最终的证据聚合与推理。
- **严密的答案后处理约束（Output Guardrails）：** 金融题目往往有极强的格式约束。我们将建立严格的后处理规则，完成多选题字母排序、去重、合法性校验，杜绝模型幻觉导致的格式扣分。

## 2. 核心架构与解题流程

系统整体架构分为三个阶段：**文档离线预处理阶段**、**在线两阶段混合检索阶段**、**智能体推理与后处理阶段**。

```tex
[原始PDF/文本] ➔ [文档解析与还原(MinerU/OCR)] ➔ [结构化分块与标签化索引]
                                                               ↓ (离线)
[题目/选项输入] ➔ [意图与关键词解析] ➔ [粗筛: 规则与关键帧匹配] ➔ [精筛: Qwen上下文压缩]
                                                               ↓ (在线)
[大模型严格推理 (Qwen)] ➔ [答案后处理约束 (字母排序/格式化)] ➔ [最终 answer.csv 输出]
```

## 3. 核心模块详细设计

### 模块一：文档预处理与高保真还原（离线阶段）

根据规则，此阶段允许使用非 Qwen 模型。金融文档包含大量跨页表格和多级条款，单纯的 `pdfplumber` 极易导致版面错乱。

1. **版面结构还原：** 采用 **MinerU** 或 **LayoutLM** 对 PDF 进行解析，精准识别出 `Heading1/2/3`、正文、页眉、页脚及复杂表格。
2. **表格高保真转化：** 将所有财务报表（financial_reports）和保险费率表中的表格统一转化为标准的 **Markdown Table** 格式，保留表头对应的行列上下文化关系。
3. **原子块（Chunking）切分规则：** * **放弃固定长度切块：** 严格按照**法律条文编号（如“第四十七条”）\**或\**合同章节**进行语义级别切块。
   - **元数据注入（Metadata Injection）：** 每一个 Chunk 头部强行注入所属文档 ID、章节标题、页码等上下文信息。例如：`[DocID: strict_csrc_035 | Title: 第四十七条] ...条文内容...`。

### 模块二：无 Embedding 纯文本多级检索（在线粗筛）

既然不能用向量，就将传统的 **BM25 算法** 与 **金融词表规则** 结合，发挥到极致。

1. **题目与选项特征提取：** 利用正则和 Qwen本地小模型（或轻量版 API）提取题目及 A/B/C/D 选项中的关键特征：**时间（如 2025年）、金额/比例（如 6.5亿元、75%）、专有名词（如 独立董事、募集资金）**。
2. **词表增强（领域知识库）：** 构建金融术语近义词表。例如，当题目出现“退保”时，检索词自动关联“解除保险合同”、“现金价值”。
3. **多级倒排检索：**
   - **第一层（文档级定位）：** A 榜直接给出 `doc_ids`，直接锁定；B 榜未提供时，利用提取的关键词和标题进行全字匹配，锁定相关联的 1~3 篇目标文档。
   - **第二层（段落级粗筛）：** 计算题目+各选项与文档各 Chunk 的 BM25 得分，融入条款编号、数值匹配的加权。每个选项召回 Top-5 相关的 Chunk，合并去重，将上下文压缩至原文档的 **5% - 10%**。

### 模块三：基于 Qwen 的动态记忆压缩与精筛（在线精筛）

将粗筛后的文本（约 1-2 万 Token）输入给 Qwen 模型的上下文窗口进行二次瘦身，目标是将输入严格控制在 **3,000 Token 以内**。

1. **证据相关性判定（Information Extraction）：** 指示 Qwen 对粗筛出来的 Chunk 进行单行打分或提取。
   - *Prompt 示例*：“请判断以下条款是否包含能够支撑或反驳选项 [X] 的证据。若有，请完整保留该条款并输出；若无，请直接回复 None。”
2. **证据聚合（Evidence Bundling）：**  将通过精筛的法条/财务数据，按照题目选项（A、B、C、D）的顺序重新拼装，剔除所有无关修饰词，合并同类项，形成一个**极致精简的“法庭证据链”上下文**。

### 模块四：Qwen 严格推理与思维链（推理阶段）

在此阶段，我们将调用主力 Qwen 模型进行最后的逻辑判断。为了防止大模型利用其自带的预训练知识“瞎猜”，必须进行**强闭环约束**。

1. **负向条件注入（Negative Prompts）：** 在 Prompt 中强调：“你是一名严谨的金融审计师。请不要使用任何你的常识或自带的外部法律知识。**如果上下文提供的证据不足以推导某选项，则该选项视为错误。**”
2. **分步评阅思维链（CoT）：**
   - 让模型对 A、B、C、D 四个选项逐一发表“评阅意见”，明确写出：该选项对应上下文中的哪一条证据（DocID + 条款名），逻辑对比是怎样的，最终该选项是 True 还是 False。

### 模块五：答案规范化与合法性检查（后处理阶段）

根据 `SPEC.md` 要求，答案若顺序不规范、包含非法字符即判错。

1. **大写规范化：** 统一将模型输出转换为大写字母。
2. **多选题强约束排序：** * 若模型识别出答案为 `["C", "A"]`，后处理脚本通过 Python 的 `"".join(sorted(set(ans)))` 自动将其校准为 `AC`。
3. **题型对齐：**
   - 如果是单选题或判断题，检查最终字符串长度是否严格为 1。若模型在多选题中输出了多个字母，确保其没有使用空格或逗号分隔。

## 4. 针对五大领域的定制化策略 (Domain-Specific Rules)

根据基线文档提示，通用的 Prompt 无法通吃所有场景，必须针对性设计：

| **领域标识 (domain)**              | **核心难点**                         | **本方案特异性后处理与提示策略**                             |
| ---------------------------------- | ------------------------------------ | ------------------------------------------------------------ |
| **insurance** (保险条款)           | 退保金、身故金的公式计算与条款比较。 | **公式显式提取：** 在预处理时用正则捕捉 `max(A, B)` 或 `已交保费 * 比例`。在 Prompt 中设置虚拟暂存器，强迫模型先列出计算公式，代入数字，最后比对选项。 |
| **regulatory** (监管法规)          | 法规施行日期、条文优先级、合规时限。 | **时限与比例强匹配：** 对 Chunk 中的“天数”、“百分比”、“不低于/不超过”进行正则高亮。模型推理时需进行时间线（Timeline）对齐。 |
| **financial_contracts** (金融合同) | 债券条款、权利义务关系、跨章节引用。 | **实体关系映射：** 识别合同主体（如：发行人、担保人、持有人）。重点检索“违约责任”、“赎回条款”章节。 |
| **financial_reports** (财务报表)   | 跨年度财务指标对比、经营表现计算。   | **跨文档年度对齐：** 由于 A 组会给连续两年的年报（如：byd_2024, byd_2025），Prompt 必须强制要求模型：“查找 2025 年报中的‘上年同期’数据，与 2024 年报中的‘本期’数据进行双向交叉核验，防止统计口径不一致。” |
| **research** (行业研报)            | 行业趋势、观点核验。                 | **语义宽松匹配：** 研报多为图表和定性描述，检索时适当放宽 BM25 的窗口大小（Window Size），允许包含研报上下文的段落进入推理。 |

## 5. Token 与成本控制指标监控

为了在线上评测的 `answer.csv` 中拿到优秀的 Token 分数，我们将实施以下工程化监控：

1. **API 级别缓存（Caching）：** 在粗筛和精筛阶段，如果多道题目关联了相同的 `doc_ids`（例如连续好几道题都在考同一款保险产品），对该文档解析出来的结构化骨架和高频 Chunk 实施**内存级缓存**，避免重复读取同一文档造成 Prompt Token 重复计费。
2. **多轮交互动态熔断：** 严格限制单道题目的最大路由轮数。若经过 2 轮检索精筛仍未找到证据，启动**兜底策略**：直接使用该文档的**核心大纲目录+已召回的最优 Chunk** 进行单轮盲推，及时止损，防止 Token 无底线消耗。
